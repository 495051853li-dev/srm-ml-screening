# 扩展到 1000+ 篇的扩展说明

本文件总结当前批量文献处理 pipeline 在从 `222` 篇扩展到 `1000+` 篇时的主要瓶颈、性能压力点和工程要求。

## 一、当前最明显的瓶颈

从当前运行结果看，最大的瓶颈不是候选筛选，而是后两层：

1. 来源获取质量
   - 很多页面是 DOI 落地页、摘要页或跳转页
   - 这些来源对实验字段抽取不友好

2. 条件与性能字段抽取能力
   - 自动抽取对催化剂身份相对友好
   - 对温度、S/C、GHSV、TOS、conversion、yield、selectivity 等字段仍偏弱

3. 期刊指标补全方式
   - 当前以本地 curated 映射表为主
   - 适合稳定运行，但不适合无限扩张

4. 当前 `analysis_ready` 门槛下导出为空
   - 说明抽取质量还不足以稳定生成分析准备子集

## 二、哪些 stage 最容易成为性能瓶颈

### `stage1_search_candidates`

风险：

- 查询量增加后，API 分页与速率限制会成为主要瓶颈
- 需要更清晰的增量更新策略，否则重复检索成本高

### `stage2_enrich_journal_metrics`

风险：

- curated 映射表越大，维护成本越高
- 如果每次都靠人工补表，扩展到 1000+ 会变慢

### `stage4_fetch_sources`

风险：

- 这是最容易成为 I/O 和网络瓶颈的 stage
- 单篇请求慢、403/404 多、重定向多
- PDF 下载和 HTML 抓取的重试成本高

### `stage5_extract_fields`

风险：

- 这是最容易成为“质量瓶颈”的 stage，而不仅是性能瓶颈
- 抓到了页面不代表能抽到条件和性能
- PDF 文本清洗和 HTML 正文识别会快速变复杂

### `stage6_qc_and_freeze`

风险：

- 行数大时问题不在计算量，而在规则是否足够稳定
- 如果 stage5 输出质量参差不齐，stage6 会持续标记大量记录为待人工复核

## 三、哪些步骤最需要缓存、断点续跑和重试机制

### 必须支持缓存与断点续跑的步骤

1. `stage1_search_candidates`
   - 缓存 API 查询结果
   - 记录查询词、查询时间、分页位置

2. `stage2_enrich_journal_metrics`
   - 缓存已补过的期刊指标
   - 避免每次重跑都重新人工补表

3. `stage4_fetch_sources`
   - 最需要断点续跑
   - 最需要失败记录
   - 最需要跳过已完成项
   - 最需要失败原因分类

4. `stage5_extract_fields`
   - 需要跳过已抽取且来源未变化的记录
   - 否则每次重跑都重复解析同一份 HTML/PDF

### 建议支持重试机制的步骤

1. `stage4_fetch_sources`
   - 对临时网络错误重试
   - 对 403/404 不做无限重试
   - 对重定向页可延后人工处理

2. `stage1_search_candidates`
   - 对 API 临时超时和 429 做有限重试

## 四、哪些步骤不应因为单篇失败而阻塞全流程

以下步骤必须坚持“单篇失败不阻塞全流程”的原则：

1. `stage1_search_candidates`
   - 单个 query 页失败不应让整个候选池构建中断

2. `stage2_enrich_journal_metrics`
   - 单个期刊找不到 IF，不应阻塞全表 enrichment

3. `stage4_fetch_sources`
   - 单篇 403/404/redirect-only，不应阻塞其余文献抓取

4. `stage5_extract_fields`
   - 单篇抽取失败，不应阻塞其余来源页解析

5. `stage6_qc_and_freeze`
   - 某些字段覆盖率差，不应阻塞 summary 和 flags 生成

## 五、从 222 扩到 1000+ 最需要补的不是哪一段代码，而是哪类能力

最优先补的是：

1. 更稳的来源层
   - 真正能拿到正文或 PDF 的比例要提高

2. 更强的实验字段抽取层
   - 尤其是条件和性能字段

3. 更清晰的缓存与重跑策略
   - 不然每次扩容都会重复做大量旧工作

4. 更稳定的 freeze 口径
   - 否则扩容后只会产生更大的“低质量 auto_draft 堆”

## 六、当前结论

这套 pipeline 已经具备“扩到更多文献”的结构基础，但距离真正支持 `1000+` 规模，还差三样关键能力：

1. 更完整的期刊指标覆盖机制
2. 更稳的批量来源获取与缓存策略
3. 更强的条件/性能字段抽取能力

也就是说，当前主风险已经不在 orchestration，而在 extraction quality。
